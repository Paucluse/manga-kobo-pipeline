## Important Fixes In This Final Batch

### RAR normalization hardening

Files:

- `src/manga_pipeline/normalizer.py`
- `tests/test_integration.py`

Behavior:

- RAR/CBR normalization now tries external tools before falling back to `rarfile`.
- If `unar` exists in the runtime image, it is preferred.
- `7z` is used when available.
- `rarfile.Error` is converted into a clear `ValueError` instead of leaking as an unexpected pipeline error.
- Partial output CBZ files are removed on failure.
- External tool output is shortened before being surfaced in error messages.

Important runtime note:

- The current Docker image does **not** install `unar`.
- Adding Debian `unar` pulled a large GNUStep dependency set and filled the Docker root filesystem during build.
- The runtime currently has `7z` and `unrar-free`.
- Some old RARs may still fail with `Unsupported Method`; the correct user-level fix is to repackage/extract them before import or install a more complete RAR extractor after expanding/cleaning Docker storage.

### Failed imports no longer count as successful imports

Files:

- `src/manga_pipeline/pipeline.py`
- `tests/test_pipeline_naming.py`

Behavior:

- Existing target CBZ files only trigger the "source must be larger" replacement guard when a successful `DONE` record owns that archive/import destination.
- Stale target CBZ files left by failed or interrupted runs are moved into `processing/replacement-backups/...` and no longer block reset/re-import.
- Directory source size now uses recursive payload size rather than the directory inode size.

Observed incident fixed:

- User extracted a 15-volume RAR set after RAR import failed.
- New directory records were generated, but they were blocked by stale archive files and reported as "source is not larger".
- The code now treats those stale archive files as failed-run artifacts, not as already imported books.
